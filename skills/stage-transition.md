# Stage Transition Skill

이 skill은 프로젝트 단계 전환 시 사용합니다.

---

## Prototype → Alpha 전환

사용자가 "Alpha로 전환하자"라고 하면:

```
Alpha 단계로 전환합니다.

지금이 프로토타입 코드를 리팩토링하고 SOLID 원칙을 적용하기 가장 좋은 시점입니다.

[프로토타입 코드 분석]
발견:
- PlayerController.cs: 200줄, 모든 로직 혼합
- GameManager.cs: 150줄, 점수/저장/UI 로직 혼합

권장사항:
1. 전체 리팩토링 (권장, 1-2주)
   - MonoBehaviour에서 로직 추출
   - SOLID 원칙 적용 (OCP, SRP)
   - 핵심 시스템 테스트 추가
   - 미래 기능을 위한 확장 포인트 생성

2. 핵심만 리팩토링 (3-5일)
   - 중요 시스템만 리팩토링
   - 스탯/스킬/아이템 시스템에 SOLID 적용
   - 기본 테스트 추가

3. 리팩토링 미루기 (권장하지 않음)
   - 현재 코드로 진행
   - Beta에서 기능 추가 어려워짐

어떤 옵션을 선호하시나요?
```

### 리팩토링 결정 후:
```
[PROJECT_CONTEXT.md 업데이트]
- 단계: Prototype → Alpha
- 리팩토링: [선택한 옵션]
- 시작일: [오늘]

앞으로:
- 핵심 시스템: TDD 40-60%
- 확장 가능 시스템에 SOLID 원칙 적용
- 모든 새 기능 깔끔하게 구현
```

## Alpha → Beta 전환

사용자가 "Beta로 전환하자"라고 하면:

```
Beta 단계로 전환합니다.

이것이 의미하는 바:
✅ 구조가 이제 잠김 - 대규모 변경 불가
✅ 추가 가능: 콘텐츠, 기존 시스템 사용하는 작은 기능
❌ 추가 불가: 구조 변경이 필요한 기능

Beta 진입 전 권장:
1. 최종 구조 리뷰 (2-3일)
2. 성능 프로파일링 및 최적화
3. 테스트 커버리지 60%+ 확보

Beta 전환을 진행할까요? (y/n)
```

### 확인 후:
```
[PROJECT_CONTEXT.md 업데이트]
- 단계: Alpha → Beta
- 구조 잠금: [날짜]
- 포커스: 버그 수정, 밸런싱, 콘텐츠 추가

앞으로:
- 새 기능 요청은 안전성 검토
- 대규모 기능은 출시 후로 권장
- 모든 변경에 테스트 필요
```

## Beta → Live 전환

사용자가 "Live로 전환하자"라고 하면:

```
Live 단계로 전환합니다.

이것이 의미하는 바:
✅ 서비스 운영 모드
✅ 모든 변경에 테스트 필수
⚠️ 리팩토링 거의 불가
⚠️ 기존 확장 포인트만 사용

Live 진입 전 권장:
1. 전체 회귀 테스트
2. 성능 최종 점검
3. 핫픽스 프로세스 준비

Live 전환을 진행할까요? (y/n)
```

---

## 단계별 정책 요약

| 단계 | 코드 품질 | 리팩토링 | TDD | SOLID |
|------|----------|----------|-----|-------|
| Prototype | 낮음 | 거의 없음 ❌ | ~10% | 최소 |
| Alpha | 중-높음 | 적극적 ✅✅✅ | 40-60% | 확장 시스템에 적용 |
| Beta | 높음 | 버그 수정만 ⚠️ | 60-80% | 기존 구조 활용 |
| Live | 매우 높음 | 거의 없음 ⚠️⚠️⚠️ | 80-100% | 확장 포인트만 사용 |

---

## Alpha 리팩토링 일정

- **Alpha 시작**: 1-2주 프로토타입 정리
- **스프린트 중간**: 스프린트당 1-2일 지속적 리팩토링
- **Alpha 종료 (Beta 전)**: 1주 최종 정리
