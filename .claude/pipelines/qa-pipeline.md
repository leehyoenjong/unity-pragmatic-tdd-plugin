# QA Pipeline

코드 또는 시스템에 대한 전체 QA 검증을 수행하는 파이프라인입니다.

## 트리거 조건

다음 패턴의 요청 시 자동 실행:
- "XX QA 해줘"
- "XX 검증해줘"
- "XX 품질 체크"
- "전체 QA 돌려줘"

## 파이프라인 흐름

```
┌─────────────────────────────────────────────────────────────┐
│                      QA Pipeline                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  [1] qa-tech ─────────────────────────────────────────────► │
│      기술적 버그 패턴 분석                                    │
│      - 경계값, Null, 동시성, 상태, 계산, 컬렉션              │
│                                                              │
│  [2] qa-security ─────────────────────────────────────────► │
│      보안 취약점 분석                                        │
│      - 어뷰징, 패킷 조작, 메모리 해킹                        │
│                                                              │
│  [3] qa-balance ──────────────────────────────────────────► │
│      밸런스 분석 (선택적)                                    │
│      - 경제 시스템, 난이도, 리텐션                           │
│                                                              │
│  [4] qa-tc ───────────────────────────────────────────────► │
│      TC 생성                                                 │
│      - 테스트 피라미드 기반 TC 문서화                        │
│                                                              │
│  [5] 종합 리포트 ─────────────────────────────────────────► │
│      전체 결과 요약                                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 단계별 설명

### Step 1: 기술 분석 (qa-tech)
```
입력: 대상 코드/시스템
출력: 기술적 버그 리포트
```

6가지 버그 패턴 + Unity 특화 체크 수행

### Step 2: 보안 분석 (qa-security)
```
입력: 대상 코드/시스템
출력: 보안 취약점 리포트
```

어뷰징 가능성, 클라이언트 신뢰 문제 분석

### Step 3: 밸런스 분석 (qa-balance) - 선택적
```
입력: 게임 시스템 (경제, 전투 등)
출력: 밸런스 리포트
```

수치 기반 시스템일 경우에만 실행

### Step 4: TC 생성 (qa-tc)
```
입력: 기능 스펙 + 발견된 이슈
출력: TC 문서
```

테스트 피라미드 구조로 TC 문서화

### Step 5: 종합 리포트
모든 분석 결과를 하나의 리포트로 통합

## 출력 형식

```markdown
# QA 종합 리포트: {{시스템명}}

## 요약
| 항목 | 발견 수 | Critical | High | Medium | Low |
|------|---------|----------|------|--------|-----|
| 기술 버그 | N | - | - | - | - |
| 보안 취약점 | N | - | - | - | - |
| 밸런스 이슈 | N | - | - | - | - |

## 즉시 수정 필요 (Critical)
1. {{이슈}}

## 권장 수정 (High)
1. {{이슈}}

## 개선 제안 (Medium/Low)
1. {{이슈}}

## TC 요약
- Unit TC: N개
- Integration TC: N개
- Functional TC: N개

## 상세 리포트
### 기술 분석
{{qa-tech 결과}}

### 보안 분석
{{qa-security 결과}}

### 밸런스 분석
{{qa-balance 결과}}

### TC 문서
{{qa-tc 결과}}
```

## 실행 옵션

| 옵션 | 설명 |
|------|------|
| `--full` | 전체 파이프라인 (기본) |
| `--quick` | 기술+보안만 (빠른 검증) |
| `--balance` | 밸런스 분석 포함 |
| `--tc-only` | TC 생성만 |

## 사용 예시

```
"인벤토리 시스템 QA 해줘"
→ qa-tech → qa-security → qa-tc → 종합 리포트

"전투 시스템 전체 QA (밸런스 포함)"
→ qa-tech → qa-security → qa-balance → qa-tc → 종합 리포트

"빠른 보안 체크만"
→ qa-tech → qa-security → 요약 리포트
```

## 주의사항

1. **순서 중요**: 기술 → 보안 → 밸런스 → TC 순서 유지
2. **컨텍스트 전달**: 각 에이전트 결과를 다음 에이전트에 전달
3. **Critical 우선**: Critical 이슈 발견 시 즉시 알림
4. **밸런스 선택적**: 수치 시스템 아니면 스킵
